# 業務改善・自動化ポートフォリオ

LINEで送られてくる非定型な業務メッセージを構造化し、実績集計や日報登録を自動化した実務ポートフォリオです。
「現場のやり方を変えずに定着させる」を設計方針とし、Google Apps Script・Cloudflare Workers・Gemini APIを使って実運用まで落とした2本のシステムを掲載しています。

| 稼働期間 | 約3ヶ月 | 約1ヶ月 |

## このリポジトリで伝えたいこと

- **曖昧な業務運用を整理し、共通構造を見つけてテンプレート化した**: 業種が異なる2つの現場でも、「非定型テキストを受け取って構造化しスプレッドシートに登録する」という骨格は同じだった。その共通部分を先に設計し、案件ごとの差分だけを実装している
- **現場のやり方を変えずに定着させた**: Googleフォームで統一しようとしたが定着しないと判断し、LINEでの報告スタイルはそのままに、受け取る側で自動解析する構成にした
- **AIを雑に使わず、ルール化できる部分と補助利用する部分を切り分けた**: 正規表現で処理できるメッセージはそこで完結させ、どうしても解析できない場合のみAIを呼ぶ。速度・コスト・精度のバランスを意識した判断
- **共通構造を横展開して短期適用した**: 日報システムのアーキテクチャをそのままMNP集計に転用し、約1週間で2本目を完成させた
- **リリース後も改善を続けている**: 運用中に再集計の必要が生じたケースに対応するなど、稼働後も継続して改善している

## 解決した業務課題

### 営業実績の集計負荷（line-mnp-tracker）

約60人の携帯ショップスタッフがLINEグループに毎日投稿する実績報告を、管理者が週末にまとめて集計していた。書き方はスタッフごとに異なり、転記ミスと毎週末2時間の集計作業が課題だった。また、業務グループに雑談が混在するLINE運用だったため、対象外メッセージを自動でスキップする仕組みも必要だった。

### 現場日報の転記負荷（line-daily-report）

建設・施工現場の管理者がLINEに毎日送る日報を、月末に手作業で転記していた。1メッセージに複数日・複数現場・複数人が含まれることがあり、月末の転記とダブルチェックで約2時間かかっていた。最初はGoogleフォームで入力を統一しようとしたが、現場スタッフの習慣を変えることが定着の障壁になると判断し、LINEのまま自動解析する方針に切り替えた。

## 共通アーキテクチャ

### システム構成

```
LINE グループ
  ↓ Webhook
Cloudflare Worker（即時 200 OK 返却）
  ↓ 非同期 fetch
Google Apps Script
  ↓
Google スプレッドシート書き込み
```

LINE Webhookは3秒以内の応答を要求するが、GASは処理によって遅延が生じることがある。Cloudflare Workerを挟んで即時応答を返し、GASへの転送を非同期で行っている。

### 処理フロー

```
Webhookイベント受信
  → 重複排除（メッセージIDで二重処理を防ぐ）
  → ルールベースパース（正規表現・構造解析）
      成功 → スプレッドシートに書き込み → ログ記録
      失敗 → AIでパース → 書き込み → ログ記録
      両方失敗 → ログに記録してスキップ
```

### 設計上の考え方

**ルールベース中心、AIは補助**
LINEメッセージは書き方が自由なため、すべてAIに解釈させることもできる。ただし、AIは応答速度・コスト・解釈のブレという問題がある。そのため、正規表現と構造解析で処理できるものはそこで完結させ、AIはどうしても解釈できなかった場合のフォールバックに限定している。

**現場の書き方を前提にした正規化**
全角・半角・スペースの混在、複数の言い回し（「残業2h」「OT2h」「+2h」）は、実際の運用でよく起きる。これを事前に正規化しておくことで、ルールの当たり率を上げている。

**運用を止めない設計**
重複排除・送信取消対応・月末遅延ケア・処理ログなど、「実際に動かすと起きる問題」を実装に織り込んでいる。

### 共通の運用設計

| 機能 | 内容 |
|---|---|
| 重複排除 | メッセージIDをログシートで照合し、同一IDの二重処理を防ぐ |
| 送信取消対応 | LINEの送信取消操作を受け取り、対応行をスプレッドシートから削除 |
| 月末自動締め | 毎月末に当月データをアーカイブして翌月用にリセット |
| 月末遅延ケア | 自動処理が翌月1日早朝にずれた場合でも前月の締め漏れを検知して実行 |
| 手動メニュー | スプレッドシートのメニューから月末締めを手動実行できる |
| 処理ログ | 全イベントの処理結果をシートに記録。障害時の原因特定に使う |

### ファイル構成（両プロジェクト共通）

| ファイル | 役割 |
|---|---|
| `webhook.js` | Webhook受信・メッセージ解析・スプレッドシート書き込み |
| `management.js` | 月末締め処理・アーカイブ・手動メニュー |
| `worker.js` | 即時応答 → GASへの転送（Cloudflare Workers） |

## line-daily-report — LINE現場日報自動登録

詳細は [`line-daily-report/README.md`](./line-daily-report/README.md) を参照。

### 背景と試行

建設・施工現場の管理者がLINEに毎日送る日報（誰がどの現場に何人工入ったか）を、月末にまとめて手作業で転記していた。最初はGoogleフォームで入力を統一する案を約1ヶ月試みたが、現場スタッフの入力習慣を変えることが定着の障壁になると判断し、LINEでの報告スタイルはそのままに受け取る側で自動解析する構成にした。

### 入出力サンプル

**入力（LINEメッセージ）**

```
3/18（月）
株式会社〇〇
渋谷現場
田中 1
鈴木 半日 残業2h
佐藤 夜勤 1
```

**出力（スプレッドシートに自動登録）**

| 日付 | 取引先 | 現場 | 職人名 | 人工 | 勤務体系 | 残業 | 判定方法 |
|---|---|---|---|---|---|---|---|
| 2026/03/18 | 株式会社〇〇 | 渋谷現場 | 田中 | 1 | — | 0h | 従来ルール |
| 2026/03/18 | 株式会社〇〇 | 渋谷現場 | 鈴木 | 0.5 | — | 2h | 従来ルール |
| 2026/03/18 | 株式会社〇〇 | 渋谷現場 | 佐藤 | 1 | 夜勤 | 0h | 従来ルール |

### この案件ならではの工夫

- **複数人・複数日への対応**: 「日付→取引先→現場→職人リスト」の階層を1行ずつ読みながら状態を管理し、1メッセージで複数日・複数現場・複数人の登録に対応
- **表記の揺れへの対応**: `半日`・`午前`・`AM`・`0.5` はすべて0.5人工として扱う。残業も `残業2h`・`OT2h`・`+2h` など複数の書き方に対応
- **AIのフォールバック利用**: ルールで解析できなかった場合のみAIを呼び、出力形式をあらかじめ指定して受け取ることで解析のズレを抑えている
- **管理者コマンド**: LINEで「管理者登録」「管理者解除」と送ることで管理者一覧を更新。登録成功時は管理者LINEに自動通知

### 成果

月末の転記（約1時間）とダブルチェック（約1時間）が不要になり、月2時間の作業を削減した。稼働後に再集計が必要になったケースが発生したが、処理ログを残していたため原因の特定と修正はスムーズだった。

## line-mnp-tracker — LINE MNP実績自動集計

詳細は [`line-mnp-tracker/README.md`](./line-mnp-tracker/README.md) を参照。

### 背景

約60人の携帯ショップスタッフが毎日LINEグループに投稿する営業実績を、管理者が週末にまとめて集計していた。毎週末2時間ほどの手作業で、転記ミスや集計タイミングのズレが課題だった。業務グループに雑談も混在するLINE運用だったため、対象外メッセージをスキップする仕組みも必要だった。

### 入出力サンプル

**入力（LINEメッセージ）**

```
【1日の店舗実績】
3/18（月）
ヤマダ電機〇〇店
田中
当日MNP 3件/計15件
当日1G光 1件
稼働数 3日
```

**出力（スプレッドシートに自動登録）**

| 日付 | 店舗 | 担当者 | MNP（当日） | MNP（月累計） | 光 | 稼働数 |
|---|---|---|---|---|---|---|
| 2026/03/18 | ヤマダ電機〇〇店 | 田中 | 3 | 15 | 1 | 3 |

### この案件ならではの工夫

- **雑談スキップ**: 「1日の実績」と「個人獲得」の両方を含むメッセージのみ処理対象とし、それ以外はスキップ
- **表記の揺れへの対応**: 全角・半角・スペースの混在（「MNP３」「mnp:3」「当日MNP 3件」など）を統一して処理
- **累計報告の優先**: 「当日MNP X件/計Y件」形式がある場合、当日件数の単純合算より月間累計（Y）を優先して集計。報告方式がスタッフによって混在しても月次ランキングが正しく出る
- **横展開による短期開発**: line-daily-reportのアーキテクチャをそのまま転用し、MNP固有のロジックだけ書き直すことで約1週間で完成

### 成果

毎週末の集計作業（週2時間）が不要になった。月次ランキングも自動生成されるため、集計後の加工作業もなくなった。

## 技術スタック・選定理由

| 技術 | 選定理由 |
|---|---|
| Google Apps Script | スプレッドシートとの親和性が高く、Webhook・Drive・スケジューラーを追加インフラなしで扱える。運用担当者がスプレッドシートで結果を確認・編集できる点も重視した |
| Cloudflare Workers | GASの応答遅延問題を回避するために導入。無料枠で動作し、運用コストがかからない |
| Gemini API (gemini-2.5-flash) | ルールで処理できなかったメッセージのフォールバック用。応答のJSON形式を指定することで出力の構造を固定している |
| LINE Messaging API | 現場スタッフがすでにLINEで報告しているため、ツールを変えずに自動化できる |
| Google Drive API | 月次アーカイブ時のフォルダ移動に使用。GASのDriveAppだけではフォルダ間移動ができないため |

## 制約と今後の改善

**現状の制約**

- 解析精度はメッセージの書き方に依存する。想定外の形式はAIフォールバックを経てもスキップになる場合がある
- GASはスプレッドシートへの書き込みが直列処理のため、メッセージが短時間に集中すると処理が遅れることがある
- 月末アーカイブはスプレッドシートごとコピーする方式のため、データ量が増えると実行時間が伸びる

**改善候補**

- スキップされたメッセージの確認・手動登録フローの整備
- パース失敗パターンを蓄積して正規表現を拡充する仕組み
